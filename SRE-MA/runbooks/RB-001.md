# RB-001: Payment Gateway Latency Spike

## What This Incident Looks Like

Payment gateway p99 latency is elevated above normal thresholds.
Users are experiencing slow or timing-out payment transactions.
This incident may cascade — high latency leads to connection pool buildup,
which then causes downstream failures in account_ledger and fraud_detector.

---

## Service Context

The payment_gateway sits between the API gateway and card_rails (Stripe/card network).
Every payment transaction touches this path:

```
api_gateway → payment_gateway → card_rails (external)
                              → fraud_detector
                              → account_ledger
```

A problem in card_rails immediately shows up as latency in payment_gateway.
A problem internal to payment_gateway shows up as errors AND latency together.

---

## Common Root Causes

### 1. Card Rails Throttling (most common)
The external card network is rate-limiting or experiencing degradation.
This shows up as 429 (too many requests) or 504 (gateway timeout) errors.
Latency spikes but error rate stays relatively low initially.
Pattern: gradual latency increase, 429/504 errors, no recent deployment.

### 2. Recent Bad Deployment
A code change introduced a regression in payment_gateway itself.
This shows up as sudden spike in both latency AND 500 errors together.
Pattern: sharp sudden onset, correlates exactly with deployment time.

### 3. Database Connection Pool Pressure
account_ledger or postgres_primary is under pressure.
payment_gateway is waiting for DB responses downstream.
Pattern: slow query logs, HikariPool warnings, latency increases gradually.

### 4. Fraud Detector Slowdown
fraud_detector ML model inference is slow, blocking payment approvals.
Pattern: fraud_model_prediction_latency_seconds metric elevated.

---

## How To Investigate

### Step 1 — Establish current severity
Query Prometheus for current p99 latency on payment_gateway.
Note whether latency is still rising, stable, or recovering.

### Step 2 — Look at error patterns in Loki
Search Loki logs for payment_gateway service.
Key things to look for:
- `card_rails timeout` or `upstream did not respond` → card rails issue
- `HikariPool` or `connection pool exhausted` → DB pressure issue
- `Internal server error` or `unhandled exception` → code bug
- `circuit breaker OPEN` → cascading failure already in progress
- Sudden spike in 500 errors matching a specific deployment time → bad deploy

### Step 3 — Check deployment history
Was there a deployment to payment_gateway in the last 30 minutes?
If yes and errors started immediately after — strong signal of bad deployment.

### Step 4 — Check card_rails specifically
Look for elevated 429 and 504 status codes from payment_gateway.
If these are high, card_rails is the external bottleneck — not internal code.

### Step 5 — Check circuit breaker state
If circuit_breaker_state metric is OPEN for payment_gateway → card_rails:
The system has already detected the downstream failure.
Fast-fail mode is active — most requests are being rejected immediately.

---

## Remediation Actions

### If card_rails is throttling:
Enable the request queuing feature flag on payment_gateway.Agent Tool: set_feature_flag(service="payment_gateway", flag="card_rails_queue_enabled", value=True)Approval Required: False (Auto-approved, reversible).After enabling, verify error rate drops within 2 minutes.

### If bad deployment caused the issue:
Roll back payment_gateway to the previous revision.Agent Tool: restart_service(service_name="payment_gateway") (Simulates a rollback in our environment)Approval Required: True (Destructive action, requires HITL gate).After rollback, verify p99 latency recovers within 90 seconds.

### If DB connection pool is the cause:
This is outside the scope of RB-001.Agent Action: Immediately route workflow to Runbook RB-004.Do not attempt payment_gateway remediation — wrong laye

### If root cause is unclear after investigation:
Agent Tool: notify_escalation(channel="incidents", summary="Root cause unclear. Data collected: ...")Do not attempt blind remediation — escalate to human on-call.

---

## What Recovery Looks Like

- p99 latency dropping back below 500ms
- 500 error rate returning below 1%
- No new circuit breaker open events
- Payment transaction success rate returning to baseline
- Recovery typically takes 2-5 minutes after correct action is taken

## What To Tell Stakeholders

Keep communication factual and time-bound:
- "Payment processing is experiencing elevated latency since [TIME]"
- "We have identified [ROOT CAUSE] and are executing [ACTION]"
- "Expected recovery in [N] minutes"
- "We will confirm resolution and share post-mortem within 24 hours"

---

## Escalation Signals

Escalate immediately to human on-call engineer if:
- Automated remediation steps did not reduce latency
- Circuit breaker has been OPEN for more than 10 minutes
- Error rate exceeds 50% — revenue impact is severe
- Multiple services are degraded simultaneously — cascading failure
- Root cause cannot be identified from available signals
