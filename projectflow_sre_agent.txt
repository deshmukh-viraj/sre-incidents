==================================================================================
       MULTI-AGENT SRE INCIDENT RESPONSE ORCHESTRATOR
       Fintech SRE Simulation Environment + LangGraph Agent System
       Full Project Workflow — End to End
==================================================================================

PROJECT OVERVIEW:
  - Simulation layer:   Generates realistic metrics (Prometheus), structured JSON
                        logs (Loki), and traces (OTel/Tempo) for a 6-service
                        fintech topology across 6 failure scenarios.
  - Agent layer:        LangGraph state-graph orchestrating 4 specialised agents
                        (Detector, Diagnoser, Remediator, Communicator) that
                        detect, diagnose, and remediate incidents autonomously
                        with human-in-the-loop gates for destructive actions.
  - API layer:          FastAPI exposing incident lifecycle endpoints.
  - Evaluation layer:   Deterministic + LLM-as-judge evaluation, MTTR dashboard.
  - Deployment:         Docker Compose (local) → ECR + EKS (production).

TECH STACK:
  Python 3.10 | LangGraph | FastAPI | OpenAI GPT-4o | Ollama (Llama3 fallback)
  FAISS | SQLite → Redis | Prometheus | Grafana | Loki | Tempo | Alertmanager
  DVC + S3 | MLflow + DagShub | Docker Compose | GitHub Actions | AWS EKS

PHASE 5 — LANGGRAPH MULTI-AGENT SYSTEM
----------------------------------------------------------------------------------
>> Build the 4 agents + orchestrator state machine.
>> Shared state object flows through: Detector → Orchestrator → [Diagnoser ||
>> Remediator || Communicator in parallel] → Human Gate → Execution.

28. Define the shared AgentState (agents/orchestrator.py):
    from langgraph.graph import StateGraph, END
    from typing import TypedDict, List, Optional

    class AgentState(TypedDict):
        incident_id: str
        raw_signals: dict           << from Prometheus/Loki/Tempo
        incident_summary: str       << Detector output
        severity: str               << SEV1..SEV5
        hypotheses: List[dict]      << [{hypothesis, evidence, confidence}, ...]
        root_cause: Optional[str]
        action_plan: List[dict]     << [{action, blast_radius, reversible}, ...]
        requires_approval: bool
        human_approved: bool
        communication_draft: str
        resolution_status: str      << open|investigating|mitigated|resolved
        mttr_seconds: Optional[int]
        active_scenario: Optional[str]

29. Build detector_agent.py:
    - Input: raw_signals dict (metric values, log excerpts, alert name)
    - Uses: Prometheus tool to get current p99, error_rate, SLO budget
    - Uses: Loki tool to get last 20 error logs for affected service
    - Output: incident_summary string + severity classification
    - Prompt pattern: "Given these signals: {signals}, classify the incident
      severity (SEV1-5) and write a 2-sentence incident summary."
    - Output schema (Pydantic): IncidentSummary(severity, summary, affected_services)

30. Build diagnoser_agent.py (most complex agent):
    - Input: incident_summary + raw_signals
    - Uses: runbook_retrieval_tool (FAISS) to get relevant runbook chunks
    - Uses: prometheus_tool, loki_tool, tempo_tool for evidence gathering
    - Runs 3 parallel hypothesis probes using LangGraph parallel branches
    - Output: hypotheses list with confidence scores
    - CRITICAL: Must output structured Pydantic schema:
      DiagnosisResult(hypotheses=[{hypothesis, evidence, confidence, alternative}])
    - Confidence gate: if max(confidence) < 0.70 → set requires_approval=True

31. Build remediator_agent.py:
    - Input: root_cause + action_plan from diagnoser
    - Uses: kubectl_tool (dry-run mode ONLY unless approved)
    - Blast radius classification logic (build as separate function):
      "pod" → auto-approve | "service" → warn | "cluster" → always human gate
    - Sets requires_approval=True for any action where reversible=False
    - Output: action_plan list with blast_radius tags

32. Build communicator_agent.py:
    - Input: incident_summary + root_cause + resolution_status
    - Generates 3 outputs simultaneously:
      a) status_page_update (customer-facing, no internal details)
      b) war_room_summary (internal Slack, technical detail)
      c) escalation_message (if severity SEV1/SEV2)
    - Runs independently of Diagnoser/Remediator (pure parallel)

33. Build orchestrator.py — the LangGraph state graph:

    graph = StateGraph(AgentState)

    graph.add_node("detector",       run_detector)
    graph.add_node("diagnoser",      run_diagnoser)
    graph.add_node("remediator",     run_remediator)
    graph.add_node("communicator",   run_communicator)
    graph.add_node("human_gate",     human_approval_gate)
    graph.add_node("execute",        execute_remediation)

    graph.set_entry_point("detector")

    graph.add_edge("detector", "diagnoser")
    graph.add_edge("detector", "communicator")    << parallel branch

    graph.add_conditional_edges("diagnoser", route_after_diagnosis, {
        "remediate": "remediator",
        "escalate":  END,            << low confidence → human escalation
    })

    graph.add_conditional_edges("remediator", check_approval_needed, {
        "needs_approval": "human_gate",
        "auto_approved":  "execute",
    })

    graph.add_edge("human_gate", "execute")
    graph.add_edge("execute",    END)

    app = graph.compile(checkpointer=SqliteSaver("data/incident_state.db"))

    >> SqliteSaver gives you persistence + replay for free (MVP)
    >> Switch to RedisSaver for production scale

34. Build human_approval_gate node:
    - Saves action_plan to SQLite pending_approvals table
    - Calls webhook: POST http://localhost:9095/approvals with action details
    - Polls every 5s for up to APPROVAL_TIMEOUT_SECONDS (from params.yaml)
    - If no response within timeout → escalate, do NOT auto-approve
    - Returns: human_approved: True/False

PHASE 9 — EVALUATION FRAMEWORK
----------------------------------------------------------------------------------
>> Two-tier evaluation: (1) deterministic checks, (2) LLM-as-judge.
>> Powers the Grafana MTTR success dashboard.

63. Build evaluation/evaluator.py with deterministic checks:

    def evaluate_resolution(result: AgentState, ground_truth: dict) -> dict:
        checks = {
            "correct_root_cause": result["root_cause"] == ground_truth["root_cause"],
            "within_mttr_target": result["mttr_seconds"] < params["evaluation"]["mttr_target_seconds"],
            "no_destructive_action_without_approval": validate_approval_gates(result),
            "communication_sent": bool(result["communication_draft"]),
            "confidence_above_threshold": result["hypotheses"][0]["confidence"] >=
                                          params["agent"]["diagnosis_confidence_threshold"],
        }
        checks["resolution_success"] = all(checks.values())
        return checks

64. Build LLM-as-judge evaluation (evaluation/evaluator.py):
    - Uses deepeval or direct OpenAI call
    - Evaluates reasoning quality of diagnosis:
      Criteria: Faithfulness (evidence supports conclusion), Completeness

BACKGROUND CONCEPTS — READ ONCE
----------------------------------------------------------------------------------

***** LangGraph vs LangChain *****
LangChain = library of LLM primitives (prompts, tools, chains).
LangGraph = state-machine framework built on LangChain for multi-step, branching,
and cyclic agent workflows. Think of LangChain as the individual functions and
LangGraph as the control flow graph that connects them. You use both.

***** Why SqliteSaver for MVP (not Redis) *****
LangGraph checkpointers persist agent state between steps so that if a step
fails, the graph can resume from the last checkpoint rather than starting over.
SqliteSaver writes to a local .db file — zero infrastructure, perfect for MVP.
Switch to RedisSaver when you need horizontal scaling or multi-instance state.

***** Why FAISS instead of Pinecone/Weaviate *****
FAISS is an in-process vector index (a library, not a server). It requires no
external service, starts in milliseconds, and is perfectly sufficient for a
few thousand runbook chunks. Pinecone/Weaviate are network-call vector databases
needed only when the index is too large for RAM or you need multi-client access.

***** CloudFormation and EKS *****
eksctl generates CloudFormation templates behind the scenes to provision:
1. eksctl-sre-agent-cluster-cluster stack → EKS control plane
2. eksctl-sre-agent-cluster-nodegroup-sre-agent-nodes → worker nodes
Each stack is a logical group of AWS resources. Always check CloudFormation
console after cluster deletion to confirm all stacks are removed — orphaned
stacks continue billing.

***** Fleet Request Quota *****
When eksctl creates a NodeGroup, AWS provisions EC2 instances via Auto Scaling
Groups which consume Fleet Request quota. If cluster creation fails with
"You've reached your quota for maximum Fleet Requests", either delete other
running ASGs/clusters or request a quota increase in AWS Service Quotas.

***** PVC in Kubernetes *****
PersistentVolumeClaim (PVC) = a request for storage in Kubernetes.
If SQLite is used as the agent state store in EKS, you need a PVC backed by
an EBS volume so state survives pod restarts. Add to deployment.yaml:
  volumeMounts: [{ mountPath: /app/data, name: agent-state }]
  volumes: [{ name: agent-state, persistentVolumeClaim: { claimName: agent-pvc } }]

***** Token Cost Management *****
During a complex incident with 3 diagnostic loops across 4 agents, token usage
can reach 100k–500k tokens per incident. Track this via the token_cost_usd
MLflow metric. If a single incident exceeds COST_THRESHOLD (params.yaml),
force-escalate to human rather than continuing agent loops. Set per-agent
max_tokens in params.yaml, not hardcoded.

***** Human-in-the-Loop Gate *****
The human gate MUST be a non-blocking async wait. The agent state is persisted
to SQLite at the gate point. The FastAPI endpoint POST /incidents/{id}/approve
resumes the graph from checkpoint. This is why SqliteSaver is essential:
without it, the graph state is lost if the process restarts during approval wait.
