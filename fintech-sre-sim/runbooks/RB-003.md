# RB-003: Data Exfiltration Alert

## What This Incident Looks Like

Anomalous API request patterns suggesting unauthorized bulk data access.
A single IP or API key is making far more requests than normal.
Authentication failures spike first, then succeed — credential stuffing pattern.
This is a security incident, not an availability incident.
No direct SLO breach but regulatory notification may be required within hours.

---

## Service Context

The attack surface is the api_gateway, which fronts all customer data:
```
attacker → api_gateway → account_ledger (customer PII)
                       → kyc_service (identity documents)
                       → payment_gateway (transaction history)
```

A successful credential stuffing attack gives the attacker
valid session tokens, allowing them to query account data at scale.
The SIEM receives all security events via syslog forwarder.

---

## Common Attack Patterns

### 1. Credential Stuffing → Bulk Export
Phase 1: High volume of auth failures from single IP (8% failure rate).
Phase 2: Attacker finds valid credentials, auth failures drop.
Phase 3: Authenticated bulk requests to account_ledger begin.
Phase 4: Rate limiting kicks in, attacker sees 429 floods.

### 2. Compromised API Key
A valid API key was leaked (GitHub, Postman, etc.).
Attacker uses it directly — no auth failure phase.
Request rate suddenly jumps 10x from unfamiliar IP.
No 401 errors — all requests are authenticated.

### 3. Insider Threat
Legitimate user account making bulk exports outside business hours.
Request pattern is authenticated but volume is anomalous.
Source IP may be internal or VPN.

---

## How To Investigate

### Step 1 — Establish the request pattern
Query Prometheus for api_gateway request rate.
Is it 10x above the hourly baseline? More?
Note the exact time the anomaly started.

### Step 2 — Identify the source IP
Search Loki security event logs for api_gateway.
Filter for events in the anomaly window.
Look for: single IP making repeated requests, unusual geo location.
Suspicious IPs often come from 185.x.x.x range (known hosting/proxy ranges).

### Step 3 — Check authentication pattern
Look at 401 error rate on api_gateway over the last hour.
High 401s followed by sudden drop = credential stuffing succeeded.
No 401s at all = compromised API key (skipped auth phase).

### Step 4 — Identify what data was accessed
Check account_ledger request logs for bulk read patterns.
Look for: GET /v1/ledger/transactions at high frequency.
Check kyc_service logs for document access patterns.
This determines regulatory notification requirements.

### Step 5 — Check SIEM correlation
Look for bulk_data_export or anomalous_transaction_volume security events.
These are already flagged with regulation and control_reference fields.
GDPR Art.32 events = PII was accessed = 72-hour notification clock starts.

---

## Remediation Actions

### Immediate containment (no approval needed):
Block the attacking IP at api_gateway level.
Revoke the compromised API key immediately.
These are reversible — IP can be unblocked, key can be reissued.

### If credential stuffing confirmed:
Force password reset for all accounts accessed from the attacker IP.
Invalidate all active sessions for affected accounts.
This impacts real users — notify them via email.

### Rate limiting escalation:
Reduce rate limit threshold for unauthenticated requests immediately.
Enable CAPTCHA challenge for login endpoints.

### Evidence preservation:
Do NOT delete or rotate logs during active investigation.
Export the relevant log window to secure storage before it ages out.
This is required for regulatory compliance and forensics.

---

## Regulatory Requirements

### GDPR (if EU customer PII was accessed):
72-hour notification window to supervisory authority starts at discovery.
Document: what data, how many subjects, what was the attacker access.
Notify affected data subjects if high risk to their rights.

### PCI-DSS (if cardholder data was accessed):
Immediate notification to card brands and acquiring bank required.
Forensic investigation required — do not alter systems.
Preserve all logs, network captures, and access records.

---

## What Recovery Looks Like

- Attacking IP blocked, request rate returns to baseline
- No new bulk_data_export security events
- Authentication failure rate returns to normal background level
- SIEM no longer generating correlation alerts

---

## Escalation Signals

Escalate immediately to security team if:
- Attacker has been active for more than 15 minutes before detection
- PII or cardholder data was confirmed accessed
- Attack is coming from multiple IPs simultaneously
- Internal IP or VPN IP is the source (insider threat)
- You cannot identify what data was accessed
