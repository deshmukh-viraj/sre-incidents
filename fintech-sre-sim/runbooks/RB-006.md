# RB-006: Fraud Model Degradation

## What This Incident Looks Like

The fraud detection ML model is producing wrong scores.
Legitimate payments are being declined at an abnormal rate.
Baseline decline rate is around 0.8% — anything above 3% needs investigation.
Above 5% is a business SLO breach — revenue impact is significant.
This incident is insidious: the model continues running, no errors appear,
metrics look healthy until you check decline rates and model precision.

---

## Service Context

The fraud detection pipeline:
```
payment_gateway → fraud_detector → redis (ML model cache)
                                 → feature_pipeline (upstream data)
```

The feature pipeline feeds real-time transaction features to the model.
If the feature pipeline silently fails, the model receives stale features.
Stale features cause the model to misclassify legitimate transactions as fraud.
This is the "silent drift" pattern — no errors, just wrong outputs.

---

## Common Root Causes

### 1. Feature Pipeline Failure (most common)
The upstream data pipeline that feeds features to the model has stalled.
Model receives features from hours ago instead of real-time data.
Scores become systematically wrong for current transaction patterns.
Pattern: model latency normal, but precision drops. No pipeline errors visible.

### 2. Model Weights Drift
The deployed model weights have not been retrained recently.
Transaction patterns have shifted but model hasn't adapted.
Pattern: gradual precision decline over days, not sudden.
Decline rate creeps up slowly rather than spiking.

### 3. Redis Cache Corruption
The model scores cached in Redis are stale or corrupted.
Cache is serving old scores for new transaction contexts.
Pattern: model prediction latency drops (serving from cache) but accuracy drops.

### 4. High Traffic Overwhelming Model Serving
Too many concurrent inference requests for the model server.
Model serving returns default scores or errors under load.
Pattern: fraud_model_prediction_latency_seconds spikes alongside decline rate.

---

## How To Investigate

### Step 1 — Confirm decline rate is actually elevated
Query payment_transactions_total with status=declined over last 10 minutes.
Divide by total transactions to get decline rate.
Compare to 0.8% baseline. Above 3% = investigate. Above 5% = urgent.

### Step 2 — Check model inference latency
Query fraud_model_prediction_latency_seconds p99.
Above 200ms = model is under pressure or serving is degraded.
Very low latency (under 10ms) = possibly serving cached/default scores.

### Step 3 — Check feature pipeline health
Look for feature pipeline metrics or logs showing staleness.
Key question: when did the feature pipeline last successfully update?
If features are more than 15 minutes old — this is the root cause.

### Step 4 — Sample declined transactions in Loki
Look at payment_gateway logs for transactions with status=declined.
Are they spread across many merchants and card types (model issue)?
Or concentrated on one merchant or card type (legitimate fraud spike)?
Broad distribution = model is wrong. Narrow = real fraud.

### Step 5 — Check fraud_score distribution
Look at the fraud_score_bucket metric distribution.
If scores are clustered unusually high — model is over-scoring everything.
If scores are bimodal with a different split than normal — drift.

---

## Remediation Actions

### If feature pipeline is stale:
Restart the feature pipeline service to force a fresh data pull.
Verify features update within 5 minutes of restart.
Monitor decline rate — should recover within 10-15 minutes.

### If model weights are drifted:
This requires the ML team to retrain and redeploy the model.
SRE action: enable the fallback rule-based fraud scorer immediately.
Fallback uses hardcoded rules — slower and less accurate but reliable.
Flag: `fraud_model_fallback_enabled`

### If Redis cache is corrupted:
Flush the fraud score cache in Redis.
Model will recompute fresh scores — brief latency spike expected.
Monitor decline rate recovery after cache flush.

### Emergency: decline rate above 10%:
Temporarily raise the fraud score threshold to reduce false positives.
This increases fraud risk slightly but stops the bleeding on revenue.
Requires approval — this is a business tradeoff decision.
Threshold config: `fraud_score_threshold` (default: 0.7, raise to 0.85)

---

## What Recovery Looks Like

- Payment decline rate returning to below 1%
- fraud_model_prediction_latency_seconds returning to normal range
- Fraud score distribution returning to historical pattern
- No customer escalations about legitimate payments being blocked

## What To Tell Stakeholders

This incident has direct revenue impact — communicate proactively:
- "Fraud model is producing elevated false positives since [TIME]"
- "Legitimate payments are being declined at [X]% vs normal [0.8]%"
- "We are [action taken] — estimated recovery in [N] minutes"
- Quantify impact: "Approximately [N] transactions affected"

---

## Escalation Signals

Escalate to ML team immediately if:
- Feature pipeline restart did not resolve the issue
- Decline rate exceeds 10% and is still rising
- Model inference latency exceeds 500ms
- You cannot determine when features were last updated
- Multiple fraud model alerts firing simultaneously
