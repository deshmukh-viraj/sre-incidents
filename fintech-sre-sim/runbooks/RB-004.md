# RB-004: Database Connection Pool Exhaustion

## What This Incident Looks Like

The Postgres connection pool is filling up beyond safe thresholds.
account_ledger transactions are slowing down or failing entirely.
Slow query warnings appear in Loki before errors do.
Left unresolved, this reaches 100% pool utilization and
new connections are rejected entirely — full service outage.

---

## Service Context

postgres_primary has a fixed connection limit:
```
account_ledger (multiple pods) → postgres_primary (max_connections)
                               → postgres_replica (read queries)
```

Each account_ledger pod holds a connection pool.
If a bug prevents connections from being returned to the pool,
the pool fills up gradually — this is the "slow leak" pattern.
High traffic can also exhaust the pool suddenly — "burst" pattern.

---

## Common Root Causes

### 1. Connection Leak in Code (most common)
A code path (usually an error handler) exits without closing the connection.
Pool fills up slowly over hours. Restarts temporarily fix it.
Pattern: gradual pool growth over time, worse after high traffic periods.
Known issue: PR #4721 — ledger debit handler error path.

### 2. Long-Running Queries Holding Connections
A slow query is holding a connection open for minutes instead of milliseconds.
Other requests queue up waiting for a free connection.
Pattern: slow query warnings in Loki, pool utilization spike tied to query start.

### 3. Traffic Burst Without Pool Scaling
Sudden traffic spike creates more concurrent requests than pool slots.
Pattern: pool exhaustion correlates exactly with RPS spike.
No slow queries — just too many simultaneous legitimate requests.

### 4. Postgres Replica Failover
Replica goes down, all read traffic redirects to primary.
Primary connection pool doubles in usage suddenly.
Pattern: replica health alert fires just before pool exhaustion.

---

## How To Investigate

### Step 1 — Establish pool utilization level
Query db_connection_pool_active / db_connection_pool_max for postgres_primary.
Note the current percentage and the rate of growth.
Growing slowly over hours = leak. Spiked suddenly = burst or failover.

### Step 2 — Check for slow queries in Loki
Search account_ledger and db_proxy logs for SLOW_QUERY alert field.
Look for queries taking more than 1000ms.
If found: note the statement_hash and operation type (SELECT vs INSERT).

### Step 3 — Check account_ledger memory
High RSS memory on account_ledger alongside pool growth = connection leak.
Objects are accumulating in memory along with unclosed connections.

### Step 4 — Check replica health
Is postgres_replica reachable and healthy?
If replica is down, all read queries are hitting primary — doubles load.

### Step 5 — Correlate with deployments
Did pool growth start after a specific deployment?
If yes — the new code introduced the leak.

---

## Remediation Actions

### If pool is below 90% and growing slowly (leak pattern):
Identify the leaking pod and restart it.
This returns its connections to the pool immediately.
It is a temporary fix — the leak will return.
Root fix requires code rollback or hotfix of the connection handling bug.

### If pool is above 90% and still growing:
Rolling restart of all account_ledger pods immediately.
This is the fastest way to release all leaked connections at once.
Requires approval — causes brief disruption to in-flight transactions.

### If caused by long-running query:
Identify and kill the blocking query on postgres directly.
Only a DBA or senior engineer should execute this.
Note the query hash for the engineering team to optimize.

### If caused by replica failover:
Fix or replace the replica to restore read traffic routing.
Until replica is back, consider temporarily reducing account_ledger replicas
to lower connection count on primary.

### Code fix (permanent):
Roll back to the version before PR #4721 was merged.
Or apply hotfix: ensure connection.close() is called in all error paths.

---

## What Recovery Looks Like

- Pool utilization dropping below 50% after restart
- Slow query warnings disappearing from Loki
- account_ledger p99 latency returning below 200ms
- No new connection rejection errors in logs

---

## Escalation Signals

Escalate immediately if:
- Pool reaches 100% and new connections are being rejected
- Rolling restart did not reduce pool utilization
- Postgres primary itself is showing signs of OOM or high CPU
- Multiple services are affected simultaneously
